# Breakthrough Dimensions
## Stage Manager Shared Reference — Skills Only

Each dimension names: what 10x better looks like here, how options typically collapse, and the proof that matters before building the whole thing.

---

## User Experience / Transformation

**What 10x better looks like:**
The user's life or work is genuinely different after using this. Not faster — different. They can do something they couldn't do before, or they experience something in a way that feels new.

**How options collapse:**
- Jumping to a UI before proving the core interaction works
- Adding features before the core transformation is felt
- Optimizing for usability before the breakthrough behavior is defined
- Building for the average user before understanding the enthusiast who will love it first

**The proof that matters:**
One real user who says "I couldn't go back to the old way." Not "this is nice" — "this changed something for me."

---

## Performance / Speed

**What 10x better looks like:**
Something that felt slow now feels instant. Or something that took hours now takes minutes. The speed difference is large enough to change behavior — not just improve it.

**How options collapse:**
- Optimizing the wrong bottleneck
- Choosing a familiar architecture that has a known speed ceiling
- Building all features at normal speed before proving the fast path works
- Caching and workarounds instead of rethinking the fundamental approach

**The proof that matters:**
A working demo of the fast path on real data, before building anything else around it.

---

## Cost / Economics

**What 10x better looks like:**
Something that was expensive is now cheap enough to use differently — more often, by more people, or in ways that weren't previously worth it.

**How options collapse:**
- Using expensive infrastructure because it's familiar
- Building for current scale instead of the scale where the economics change
- Picking managed services that work now but price out the model at growth
- Ignoring the unit economics of the core action until it's too late to change

**The proof that matters:**
A cost model at 10x and 100x current usage that still works. Built before the architecture is locked in.

---

## Reliability / Trust

**What 10x better looks like:**
Users trust this system in situations where they couldn't trust anything before. It works when it matters most. It fails gracefully when it doesn't.

**How options collapse:**
- Adding reliability features as an afterthought
- Choosing architectures that are reliable at small scale but fragile at real scale
- Optimizing for the happy path before designing for failure
- Building trust through marketing instead of through actual behavior

**The proof that matters:**
A failure scenario test. What happens when the most important thing goes wrong — and does the system behave in a way the user can trust?

---

## Intelligence / Insight

**What 10x better looks like:**
The system knows something the user didn't know, or figures something out they couldn't figure out themselves. It gets smarter over time. It surfaces what matters.

**How options collapse:**
- Using AI as a feature instead of as the core mechanism
- Defaulting to generic models without thinking about what specialized intelligence would change
- Building data collection before defining what insight you're actually trying to generate
- Shipping a smart feature that doesn't actually change what the user does

**The proof that matters:**
One insight the system generated that surprised a real user and changed what they did next.

---

## Accessibility / Reach

**What 10x better looks like:**
People who couldn't use this kind of tool before can now. The barrier — cost, complexity, language, ability, device — is removed or dramatically lowered.

**How options collapse:**
- Building for the easiest user first, hardest user never
- Treating accessibility as a compliance requirement instead of a design constraint
- Assuming desktop and fast internet
- Internationalizing after the fact

**The proof that matters:**
A real session with someone who represents the newly reached user — not the already-comfortable one.

---

## Integration / Interoperability

**What 10x better looks like:**
This fits into how people already work instead of asking them to change. It connects the things that are currently disconnected. It makes the whole greater than the sum of the parts.

**How options collapse:**
- Building in isolation before proving the integration is valuable
- Using proprietary data formats that make connection hard later
- Optimizing the standalone experience before proving the connected experience
- Treating integrations as phase two

**The proof that matters:**
One real workflow that crosses system boundaries and works end-to-end — before building the standalone features around it.

---

## Scalability / Infrastructure

**What 10x better looks like:**
Something that broke or got expensive at scale now handles growth gracefully. Or something that required specialized infrastructure can now run anywhere.

**How options collapse:**
- Choosing familiar patterns that have known scaling ceilings
- Optimizing for developer convenience instead of operational simplicity
- Building for current load before stress-testing the architecture
- Picking managed services before understanding the cost cliff at scale

**The proof that matters:**
A load test or cost model at the scale where the breakthrough matters — run before the architecture is finalized.

---

## Privacy / Data Sovereignty

**What 10x better looks like:**
Users own their data. The system works without surveillance. Trust is structural, not promised.

**How options collapse:**
- Defaulting to centralized data storage because it's easier
- Choosing analytics and tracking tools before considering what data leaves the user's control
- Treating privacy as a compliance layer added at the end
- Building community trust features on top of a surveillance architecture

**The proof that matters:**
A data flow diagram showing exactly what leaves the user's device or account — reviewed with a real user before building.

---

## Community / Network Effects

**What 10x better looks like:**
The more people use it, the better it gets for everyone. Value compounds. The community itself becomes part of the product.

**How options collapse:**
- Building individual features before proving the network dynamic works
- Designing for single users before designing for interaction
- Treating community as a marketing channel instead of a product layer
- Building the platform before the community has anything to do on it

**The proof that matters:**
One interaction between two users that neither could have had alone — before building the infrastructure around it.
